# HTML → Figma 이관 Skill

웰체크 HTML 프로토타입(`m2choo/pharmaweb-prototype`)을 Figma 마스터 가이드 디자인 시스템 기준 화면으로 이관하는 범용 방법론. 이 문서는 재사용 가능한 프로세스만 다루며, 웰체크 전용 설정값(파일 키, 경로 등)은 이 Skill을 호출하는 쪽(Project 설정)에서 매번 주입받는다.

---

## 핵심 원칙: 시각 요소는 시스템, 콘텐츠는 프로토타입

이 Skill 전체를 관통하는 가장 중요한 원칙이며, 아래에 나오는 색상(2절)·컴포넌트(2-4절)·레이아웃(1-2절) 관련 규칙은 모두 이 원칙의 각론이다.

- **눈에 보이는 시각 요소** — 색상, 컴포넌트, 레이아웃/구조 — 는 항상 **기존 웰체크 디자인 시스템(마스터 가이드)의 규칙**을 따른다. HTML 프로토타입의 CSS 값(색상 hex, 레이아웃 수치)을 그대로 가져오지 않는다.
- **콘텐츠(알맹이)** — 텍스트, 데이터 값, 라벨, 상태값, 메뉴 항목 등 실제 내용 — 은 항상 **HTML 프로토타입 원본**을 따른다. baseline의 더미 데이터를 남겨두지 않는다.
- **mainComponent는 레이아웃(패딩, 마진, 폰트, 색상, 간격) 참조용이다. 이관 작업에서 절대 수정하지 않는다.** HTML은 새로운 기능이 추가될 때마다 변경된다. HTML에 변경사항(새 메뉴 항목, 새 컬럼, 새 상태 등)이 생기면 mainComponent를 수정하는 게 아니라, 해당 화면의 인스턴스를 `detachInstance()`하고 HTML 기준으로 콘텐츠를 채운다. 수정이 디자인 시스템에도 반영되어야 한다고 판단되면 → code-based Description 예외 케이스에 명시하고 담당자에게 넘긴다.

이 세 원칙이 충돌하는 것처럼 보일 때의 우선순위: 시각 요소 판단 → 시스템, 콘텐츠 판단 → 프로토타입, 컴포넌트 수정 여부 → 절대 수정 안 함.

## 0. 시작 전 확인 사항

### 질문 방식

한 번에 나열해서 묻지 않는다. 아래 항목을 하나씩 순차적으로 확인하고, 답을 받은 다음 다음 항목으로 넘어간다.

### 매 작업 시작 시 사용자에게 확인하는 항목

1. **HTML 소스 접근 경로** — 어느 저장소/파일/URL인지
2. **배치할 Figma 페이지/노드** — 작업 중 임시로 둘 위치
3. **(선택) 추가 참고 자료** — 웰체크 기본 디자인 시스템(Project 고정값)은 항상 참조하니 별도로 묻지 않되, 이번 작업에 한해 추가로 참고할 Figma 화면/컴포넌트가 있으면 링크를 받는다. 없으면 넘어간다.
4. **(선택) 완성 후 이관할 목적지** — 최종적으로 옮겨질 마스터 가이드 하위 위치. 미정이면 임시 위치에 두고 이관 결정은 나중으로 미룬다.

1, 2번이 비어 있으면 임의로 시작하지 않고 먼저 확인한다. 3, 4번은 선택 항목이므로 없어도 진행 가능.

### Project 고정값 (매번 묻지 않음)

**참고할 baseline** — 웰체크 프로젝트에서는 항상 고정: 마스터 가이드 파일(`Hg2fE9R6yId239HK6vTau8`)이 기준. 화면·컴포넌트를 찾을 땐 1-2절의 5단계 우선순위를 그 안에서 적용한다. 사용자가 매번 baseline을 지정할 필요 없음 — 다른 프로젝트에 이 Skill을 재사용할 경우에만 이 값을 교체한다.

**라이브 웹** — 1-2절 4순위(마스터 가이드·프로토타입 코드 둘 다로 판단 불가할 때) 참고 대상이나, 로그인이 필요해 Claude가 직접 접근 불가. URL을 알아도 자동 확인은 못하므로, 필요 시점마다 사용자에게 해당 화면 스크린샷을 요청하는 방식으로 진행한다.

## 1. 전체 구조 원칙

### 1-1. 임시 작업 → 최종 이관 (B안)

완성된 화면은 잡동사니(임시) 페이지에 두지 않는다. 마스터 가이드의 baseline 페이지 하위에 새 페이지를 만들어 그곳으로 최종 이관하는 구조로 간다.

- 작업 중: 워킹 파일의 임시 페이지에서 반복 수정
- 완료 후: 마스터 가이드 파일의 baseline 페이지 하위 새 페이지로 이동

### 1-2. 마스터 가이드 참고 우선순위 (5단계)

화면을 만들 때 아래 순서로 참고한다. 상위 단계에서 답이 나오면 하위 단계로 내려가지 않는다.

1. **동일 화면이 마스터 가이드에 이미 있으면** → 복제(`.clone()`)해서 시작
2. **동일 화면은 없지만 유사 목적 화면이 있으면** → 그 화면의 톤(색상, 폭, 여백 등)을 실측해서 차용
3. **컴포넌트 단위로만 겹치면** → 해당 컴포넌트 속성만 실측해서 재사용
4. **마스터 가이드에도 없고 HTML 프로토타입 코드만으로 시각 규칙을 확정할 수 없으면** → 라이브 웹(실제 운영 중인 서비스) 확인이 필요하다고 판단하고 **사용자에게 해당 화면 스크린샷을 요청**한다. 라이브 웹은 로그인이 필요해 직접 접근 불가 — URL을 알아도 Claude가 자체적으로 브라우징할 수 없으므로, 매번 스크린샷으로 받아 확인한다. 프로토타입 코드의 값(예: 태그 색상 클래스)이 실제 운영 규칙과 다를 수 있음 — 실측 사례: HTML은 특정 질환명에 `tag-purple`을 하드코딩해뒀지만, 실제 라이브 웹은 "당뇨/당뇨전단계=보라, 고혈압=주황, 그 외 전부=회색" 규칙으로 동작 중이었음
5. **위 네 곳 어디에도 없으면** → 반드시 사용자에게 확인 후 진행 (임의 추정으로 신규 생성 금지)

**우선순위가 서로 충돌할 때**: 시각 요소(색상/컴포넌트 스타일)는 "마스터 가이드 > 라이브 웹 > HTML 프로토타입 코드" 순으로 신뢰한다 — HTML 프로토타입은 데모용으로 작성되어 실제 운영 규칙과 다를 수 있기 때문. 콘텐츠(텍스트·데이터 값 자체)는 핵심 원칙대로 항상 HTML 프로토타입을 따른다 — 이 우선순위는 시각 규칙 판단에만 적용되고 콘텐츠 판단에는 적용되지 않는다.

> 주의: 참고 대상이 웰체크 마스터 가이드가 아닌 타 제품 파일(예: 의사웹)일 경우, 구조/레이아웃 참고는 가능하지만 **색상값은 그대로 가져오지 않는다** (2절 참고). 실측 결과, 이름이 같은 의미의 스타일이라도 실제 RGB 값이 다를 수 있음이 확인됨 [확실].
>
> 의사웹 Component Style Guide (`YRWyzc9i5g82n3qhMTkBpn`, node `3866-1804`): 약사웹은 의사웹과 컴포넌트 구조·레이아웃 패턴을 공유하므로, 컴포넌트 배치·여백·계층 구조 판단 시 참조 가능. 단, 색상 토큰은 의사웹 Semantic/* 체계 기반이므로 그대로 가져오지 않는다 — 색상은 반드시 마스터 가이드 Paint Style로 매핑할 것 (2-2절).

> 주의 (콘텐츠 일관성): 구조를 baseline에서 복제해도 텍스트 콘텐츠는 전체를 실제 프로토타입 데이터로 교체한다. 핵심 원칙 참고.

### 1-4. 화면 라벨 필수 부착

모든 화면 프레임 위에는 "화면번호\_화면명" 라벨을 캔버스 상에 붙인다. 이미 마스터 가이드·워킹 파일 양쪽에서 쓰이던 기존 관행(`Group 1000004314`~`4317` 등)과 동일한 패턴이므로, 새로 디자인하지 않고 기존 그룹을 복제해서 텍스트만 바꾼다.

**구성** [확실], 근거: 워킹 파일(`ICLchS9dDEoct2rVWswLjx`) node `365:1859` 실측
- 텍스트: "화면번호\_화면명" (예: `01_고객 목록`), 폰트 `Noto Sans KR Bold`, 크기 `100`, 색상 흰색 `rgb(255,255,255)`
- 하단 라인: 흰색, 두께 `10`, 텍스트 폭에 맞춰 배치
- 배경 사각형 없음 — 캔버스(페이지) 배경 위에 바로 얹는 형태이므로 프레임 바깥, 화면 프레임 위쪽에 위치

**작업 방법**: 새로 그리지 말고 기존 라벨 그룹(`Group 1000004314` 계열)을 `.clone()`한 뒤 텍스트만 교체한다 (1-5절 "복제 우선" 원칙과 동일).

**적용 범위**: 메인 화면뿐 아니라 6절의 모달 하위 화면(`01-1-a` 등)에도 각각 라벨을 붙인다 — 화면번호 표기 그대로 사용.

### 1-6. 공통 컴포넌트(LNB/헤더 등)의 콘텐츠 추가 처리

핵심 원칙 세 번째 항목의 적용 사례. HTML에 메뉴 항목 등 새 콘텐츠가 추가되어 있어도 mainComponent는 수정하지 않는다 — `detachInstance()` 후 이 화면에만 콘텐츠를 채운다. 디자인 시스템 반영 필요 여부는 Description 예외 케이스에 명시하고 담당자에게 넘긴다.

### 1-7. Figma 작업 패턴

- 새 프레임을 처음부터 만들기보다 기존 프레임 복제 방식이 스타일·구조 일관성 면에서 안정적
- Auto-layout 내 자식 노드 y좌표는 직접 지정 불가 — 부모 컨테이너 리사이즈로 간접 조정
- 크로스파일 프레임 복제는 Plugin API로 불가 — 디자인 가이드 파일 프레임은 수동 복사·붙여넣기 필요

## 2. 색상 처리 원칙

### 2-1. 색상 시스템 현황

웰체크 마스터 가이드 파일의 컬러 체계는 **Paint Style**로 정의되어 있음 [확실], 근거: `figma.getLocalPaintStylesAsync()` 조회 결과 `Brand Color/*`, `Body Color/*`, `Line/*`, `Point/*` 총 20개 스타일 확인. Variable Collection은 사실상 미사용(1개 컬렉션, 변수 1개) [확실].

### 2-2. 색상 결정 원칙

웰체크 확정 컬러 시스템(마스터 가이드의 20개 Paint Style)이 항상 1순위. 해당 의미의 색이 이미 있으면 신규 배정하지 않는다.

**타 제품(의사웹 등) 참고 시 주의** [확실], 근거: 의사웹 baseline 노드 2개 실측 결과, 사용된 4개 스타일 중 1개만 마스터 가이드 값과 정확히 일치(흰색), 나머지 3개(텍스트 진한색 등 사용 빈도 상위 항목 포함)는 이름이 비슷해도 RGB 값이 다름. 즉 "이름이 같은 의미니까 값도 같을 것"이라는 가정은 성립하지 않음 — 타 파일 참고 시 반드시 실측 대조 후 사용.

해당 의미의 색이 마스터 가이드에 없을 때만 HTML CSS 값을 참고해 신규 배정하고, `[추정]` 표기 후 팀 확인을 거친다.

### 2-3. 색상 바인딩 방식

raw RGB 값 하드코딩 금지, **스타일 ID 연결** 방식을 쓴다.

```js
// 1. 마스터 가이드 Paint Style 목록에서 매칭되는 스타일 찾기
const paintStyles = await figma.getLocalPaintStylesAsync();
const matched = paintStyles.find(s => s.name === "Brand Color/Basic");

// 2. 매칭되면 스타일 ID로 바인딩 (raw color 대입 금지)
if (matched) {
  await node.setFillStyleIdAsync(matched.id);
} else {
  // 3. 매칭 안 되면만 raw 값 임시 적용 + 표기
  node.fills = [{ type: 'SOLID', color: { r, g, b } }];
  // → Description/작업로그에 "[추정] 신규 컬러, 기존 체계에 없어 HTML 값(#xxxxxx) 임시 적용" 명시
}
```

**이렇게 하는 이유**: raw 값으로 복사하면 이관된 화면과 마스터 가이드 스타일 간 연결이 끊겨, 이후 브랜드 컬러가 바뀔 때 이관된 화면을 일일이 다시 찾아 수정해야 한다. 스타일 ID로 연결해두면 마스터 가이드 값 변경 시 자동 반영된다.

### 2-4. 신규 UI 요소 생성 전 필수 검색 (배지/태그/칩 등)

배지, 태그, 칩, 상태 표시처럼 반복되는 작은 UI 요소를 만들어야 할 때, `createFrame`/`createText`로 바로 만들지 않는다. 실측 사례: 태그 컬럼에 점(•) 마커가 포함된 기존 "Tag_Heavy" 컴포넌트가 있었는데, 검색 없이 플레인 프레임으로 새로 만들어 점 마커가 빠진 채 이관됨 — 시각적으로 기존 시스템과 어긋난 결과가 나왔음.

**절차 (반드시 순서대로)**
1. 현재 화면 안에 이미 쓰이고 있는 유사 요소가 있는지 먼저 확인 (예: 같은 테이블의 다른 컬럼에 이미 태그/배지가 있는가)
2. 있으면 그 컴포넌트의 `mainComponent`를 추적해서 COMPONENT/COMPONENT_SET 여부와 variant 목록을 확인
3. `component.createInstance()`로 인스턴스를 만들고, 텍스트만 교체 — 구조(닷 마커, 패딩, 폰트)는 그대로 유지
4. 필요한 색상 조합이 기존 variant에 없으면(예: 3색 체계인데 컴포넌트엔 2색만 있음), 구조는 그대로 복제하되 색상만 신규 값으로 교체하고 `[추정]` 표기
5. 화면 안에 유사 요소가 전혀 없으면 → 마스터 가이드의 "컴포넌트 모음" 페이지에서 이름으로 검색 (예: "칩", "배지", "태그")
6. 검색해도 없으면 → 이때만 새로 만든다. 단, 이 경우도 완전히 새로운 의미의 요소인지(2-2절 색상 원칙과 동일한 논리) 확인 후 진행

컴포넌트 모음 페이지 위치: 마스터 가이드 파일(`Hg2fE9R6yId239HK6vTau8`) 페이지 "컴포넌트 모음"(`457:5954`).

### 2-4-1. 컴포넌트 재사용 및 네이밍 원칙

**인스턴스 레이어 이름만 바꾸는 방식은 쓰지 않는다.** 개발자가 Dev Mode Inspect에서 실제로 보는 건 레이어 이름이 아니라 `mainComponent` 정보이므로, 레이어명만 바꾸면 개발자에게 잘못된 정보를 전달하게 됨 [확실], 근거: `Popup` 인스턴스의 레이어명은 "Popup"이지만 `mainComponent.name`은 "Property 1=Variant8".

**mainComponent 수정 금지**: 핵심 원칙 세 번째 항목 그대로 — 이관 작업에서 mainComponent는 절대 수정하지 않는다. 약사웹 전용으로 새로 필요한 컴포넌트라면 실제 새 COMPONENT(또는 기존 COMPONENT_SET에 새 Variant)로 등록 (`figma-generate-library` Skill 참고).

**네이밍 컨벤션**: 코드 컴포넌트명과의 매칭 규칙은 없음 [추정], 근거: 개발팀 직접 확인 전. 기존 "컴포넌트 모음" 페이지 관행(최상위 컴포넌트명은 한글 명사, Variant 속성은 Figma 기본값 그대로)을 따른다. Code Connect는 현재 플랜에서 비활성화 상태 [확실].

### 2-5. 크로스파일 스타일 참조 시 주의

워킹 파일에서 마스터 가이드 파일의 스타일을 직접 참조하려면 마스터 가이드가 팀 라이브러리로 게시되어 있어야 함 [불확실] — 게시 여부 확인 필요. 게시가 안 되어 있다면, 필요한 스타일의 ID 문자열을 Project 설정값으로 미리 확보해두는 우회 방식을 검토한다.

## 5. code-based Description 작성 포맷

HTML 프로토타입 코드를 기반으로 기능 Description 초안을 작성할 때의 포맷. 참고 포맷: 마스터 가이드 파일(`Hg2fE9R6yId239HK6vTau8`) node `15899:5197` PRESCRIPT-001 스타일.

**구성**
1. 화면 번호 / 화면명 / 진입경로 (기본정보)
2. 번호 리스트로 컴포넌트별 동작 서술
   - 요소명, 타입, "→ 화면번호 이동" 또는 "→ 팝업 오픈" 화살표 표기
   - 각 요소 하위에 `[제안] 프론트: ~`, `[제안] 백엔드: ~` 형태로 코드에 없는 고려사항 제안
3. 하단 예외 케이스는 불릿으로 정리

**포함하지 않는 것**: `[확실]/[근거]` 표기, 담당자, 리스크 항목 (이 Description 문서 자체의 목적과 무관)

화면 위 번호 마커(`makeMarker`)와 Description의 번호는 반드시 1:1로 매칭한다.

## 6. 다중 스텝 모달 및 다중 진입/분기 시나리오 처리 원칙

한 화면에서 FAB/버튼 클릭 시 여러 단계(step)로 구성된 모달이 뜨고, ①여러 화면에서 서로 다른 시작 지점으로 그 모달을 공유하거나, ②같은 진입점 안에서도 상태값에 따라 스텝 내부 내용이 갈라지는 경우의 처리 규칙. 두 축은 서로 다른 문제이므로 구분해서 판별한다.

### 6-1. 판별 (사용자가 알려주지 않아도 매번 먼저 수행)

이 판별은 사용자가 "여기 분기 있어"라고 알려줬을 때만 하는 게 아니라, **모달/다단계 화면을 다루는 모든 작업에서 Figma 작업 시작 전 항상 먼저 수행**한다. 기획자가 매번 분기 구조를 설명해줄 거라고 가정하지 않는다 — 소스 코드 자체가 근거이므로, 코드를 읽고 스스로 분기 구조를 재구성하는 게 원칙이다.

절차:
1. 모달/화면 관련 JS에서 `step`, `render` 계열 함수명을 먼저 grep해서 전체 함수 목록을 확보한다
2. 각 렌더 함수 내부에 `if/else`, 삼항 연산자, 상태 플래그(`isNew`, `selected*`, `has*`, `status`, `type` 등 네이밍 패턴) 기반 분기가 있는지 확인한다
3. 분기가 있으면, 그 분기가 어느 step까지 이어지는지 추적한다 (한 step의 분기가 이후 step 렌더링에도 영향을 주는 경우가 많음 — 6-3절 실제 사례처럼 Step1 분기가 Step3·4까지 이어질 수 있음)
4. 분기별로 실제 화면에 어떤 차이가 나는지(필드 유무/문구 전체 교체/버튼 동작 차이 등) 정리한 뒤에만 6-2·6-3절의 번호 부여·프레임 분리 여부를 결정한다

이 절차를 생략하고 화면에 보이는 것만 보고 이관하면, 코드에만 존재하고 기본 상태에서는 안 보이는 분기(예: 신규 고객 흐름처럼 특정 입력을 해야만 나타나는 상태)를 놓치게 된다. 반드시 코드 우선으로 확인한다.

### 6-1-1. 판별 방법 참고

- 모달이 몇 개의 step으로 구성되는지 (`step`, `현재step` 등 키워드)
- **진입점 축**: 모달을 여는 함수가 몇 종류인지, 각 함수가 어느 step부터 시작하는지 (예: `openModal()`은 Step1부터, `openModalAtStep2(data)`는 Step2부터), 어느 화면(들)이 각 함수를 호출하는지
- **상태 분기 축**: 특정 상태 플래그(예: `isNew`, `selectedCustomer` 유무)에 따라 이후 step에서 표시되는 필드·문구·버튼 동작이 달라지는지. `if (state) {...} else {...}` 형태로 렌더링 함수 안에 분기가 있는지 확인

### 6-2. 화면 번호 부여 방식

모달은 중복 제작하지 않고 **하나의 모달 묶음**으로 한 번만 만들고 여러 화면/시나리오에서 참조한다.

- 진입 화면: 기존 번호 그대로 (예: `01_약국고객리스트`)
- 모달 각 step: 진입 화면 번호에 하이픈으로 하위 번호 부여 (예: `01-1`, `01-2`, `01-3`, `01-4`)
- **상태 분기가 있는 step만** 하위 번호에 알파벳을 추가로 붙여 분리 (예: `01-1-a` 기존 고객 매칭, `01-1-b` 신규 고객 직접 추가). 분기 없는 step은 알파벳 없이 그대로 둔다

### 6-3. 상태 분기, 별도 프레임 vs 주석 판단 기준

모든 분기를 다 프레임으로 쪼개면 화면 수가 과도하게 늘어난다. 아래 기준으로 판단한다.

- **별도 프레임으로 분리**: 필드/블록 자체가 있다 없다 수준으로 다른 경우 (예: 전화번호 입력란 유무, 안내 문구 블록 전체 유무) — 개발자가 놓치면 안 되는 구조적 차이이므로 반드시 분리
- **하나의 프레임 + Description 주석**: 텍스트 한두 단어만 바뀌는 수준 (예: 버튼 라벨만 다름) — Description에 `[조건부]` 표기로 충분

**실제 사례 (customer_list_v3.html → modal_send_v2.html)**
- Step1: 검색 결과 매칭(`selectCustomer()`, 생년월일·전화번호·방문이력 있음) vs 미매칭 직접추가(`addNewCustomerByName()`, `isNew:true`, "정보 없음") → **구조 차이 큼, 프레임 분리** (`01-1-a`, `01-1-b`)
- Step2: 처방 선택 — 분기 근거 없음(공통) → 프레임 하나 (`01-2`)
- Step3: `isNew`일 때만 "신규로 추가된 고객은 아직 처방 이력이 없습니다" 안내 블록 추가 표시 → **블록 유무 차이, 프레임 분리** (`01-3-a` 기존, `01-3-b` 신규)
- Step4: `isNew`일 때 전화번호 입력란 추가 노출 + 이전 버튼 동작 다름(`goBackToStep1` vs Step2로 복귀) + 최종 발송 문구 다름 → **필드·동작 차이 큼, 프레임 분리** (`01-4-a`, `01-4-b`)

### 6-4. code-based Description 처리

진입경로·상태 분기가 있는 모달은, Description에서 분기 조건을 명시한다.

- 진입경로: "약국 고객 리스트 화면 FAB 클릭 → 팝업 오픈 (`01-1` Step1부터 진입)"
- 상태 분기: "`01-1-a` 검색 결과 매칭 시 → 기존 고객 정보로 헤더 구성 / `01-1-b` 미매칭 시 → 신규 고객 직접추가 흐름으로 분기"

모달 자체의 Description은 한 번만 작성하고, 진입 화면 쪽에서는 "어느 step·어느 분기로 들어가는지"만 참조하는 방식으로 중복 작성을 피한다.

### 6-5. 작업 범위 밖 진입점 처리

이번 이관 대상이 아닌 화면에서도 같은 모달을 호출하는 경우(예: 지금은 `customer_list`만 이관하는데 `prescription_list`도 같은 모달을 씀), 그 화면 자체는 만들지 않되 **예외 케이스 불릿에 존재를 명시**한다. 이렇게 해야 나중에 그 화면을 이관할 때 모달을 중복 제작하지 않고 재사용할 수 있다.

- 예: "※ 이 모달은 `prescription_list` 화면에서도 Step2 진입으로 재사용됨 (해당 화면 이관 시 신규 제작 없이 이 모달 참조)"

## 7. 작업 순서 요약

1. 시작 전 확인 항목 순차 확인 (0절)
2. HTML 소스에서 필요한 구간만 grep/sed로 추출 — 테이블/모달 등 콘텐츠 영역뿐 아니라 **LNB, 헤더 등 공통 컴포넌트 항목도 반드시 포함해서 대조** (기존 시스템 컴포넌트를 그대로 재사용해도 콘텐츠 차이가 있을 수 있음)
3. 마스터 가이드 참고 우선순위 5단계에 따라 기준 화면/컴포넌트 결정 (1-2절)
4. 다중 스텝 모달의 진입점 분기·상태 분기 여부 판별 (6절) — 해당되면 화면 번호를 하위 번호(+분기 알파벳) 체계로 분리
5. 색상은 스타일 ID 매칭 우선, 없으면 임시 raw + `[추정]` 표기 (2절)
6. 배지·태그 등 반복 UI 요소가 필요하면 새로 만들기 전 기존 컴포넌트부터 검색 (2-4절)
7. 확정 스니펫으로 반복 요소 조립 (4절)
8. 재사용하는 노드가 INSTANCE면 mainComponent 확인 후 rename 여부 판단 (2-4-1절)
9. 화면에 번호 마커 배치 + code-based Description 작성, 1:1 매칭 확인 (5절)
10. 작업 완료 후 마스터 가이드 baseline 페이지 하위 새 페이지로 최종 이관 (1-1절)

## 8. 컬럼 삽입 시 공간 침범(오버플로우) 처리

고정폭 컬럼 사이에 새 컬럼을 삽입하면, flex(가변폭)로 남는 공간을 채우던 기존 컬럼이 그만큼 좁아진다. 전체 폭이 화면/컨테이너 제약(예: 1920px 데스크톱 레이아웃) 안에 있는 상태에서 폭을 단순히 늘리면 테이블 전체가 화면 밖으로 넘친다.

### 8-1. 판별

컬럼을 새로 삽입하기 전, 다음을 확인한다.
- 삽입할 행/헤더의 부모 컨테이너가 `primaryAxisSizingMode: FIXED`인지 (고정폭이면 자식이 늘어난 만큼 flex 자식이 줄어듦)
- 어느 자식이 `layoutSizingHorizontal: FILL`(flex-grow)인지 — 이 컬럼이 압박받는 대상
- 압박받는 컬럼의 실제 콘텐츠 폭(예: 배지/태그 여러 개 합산 폭)이 줄어든 가용 폭보다 큰지 삽입 후 반드시 재계산해서 확인

### 8-2. 처리 원칙

**행 전체 폭을 늘리지 않는다** — 화면 레이아웃 제약을 벗어나 테이블 전체가 넘치는 더 큰 문제가 생긴다.

대신 HTML 프로토타입에 실제로 콘텐츠 오버플로우 처리 방식이 정의돼 있으면 그대로 따른다 (핵심 원칙 — 콘텐츠 관련 동작은 프로토타입 기준). 흔한 패턴: `flex-wrap: wrap`으로 다음 줄로 줄바꿈.

**확인 필요 여부 (9절 기준 적용)**: HTML에 처리 방식이 명시돼 있으면(예: CSS에 `flex-wrap`, `overflow`, "+N개 더보기" 로직 등) 확인 없이 그대로 재현한다 — 이미 정해진 원칙(프로토타입 우선)의 적용이기 때문. 반대로 HTML에도 처리 방식이 없는 경우(예: 그냥 방치돼 있거나, 실제 데이터가 프로토타입 작성 당시보다 많아진 경우)는 임의로 디자인 판단(줄바꿈/축약/툴팁 등)을 내리지 않고 사용자에게 확인한다 — 이건 새로운 판단이 필요한 영역이기 때문.

Figma 재현 방법:
```js
tagCell.layoutWrap = 'WRAP';          // 줄바꿈 허용
tagCell.counterAxisSizingMode = 'AUTO'; // 셀 높이를 내용에 맞춰 자동 확장
// 같은 행의 다른 형제 셀은 높이 고정 유지 → 늘어난 행 안에서 중앙 정렬 유지
for (const c of row.children) {
  if (c !== tagCell && 'layoutSizingVertical' in c) c.layoutSizingVertical = 'FIXED';
}
row.counterAxisSizingMode = 'AUTO'; // 행 높이도 자동 확장, 늘어난 셀 높이를 따라감
```

### 8-3. 적용 범위

모든 행에 일괄 적용하지 않는다. 콘텐츠 폭을 실제로 계산해서(8-1) 오버플로우가 나는 행에만 적용한다 — 대부분의 행은 원래 폭에 여유가 있어 손댈 필요가 없다.

## 9. 확인을 구할 때 vs 바로 진행할 때

Skill을 쓰는 목적 자체가 반복 확인을 줄이는 것이므로, 이미 이 문서에 원칙이 정해진 작업은 매번 승인을 구하지 않고 바로 진행한다.

**바로 진행 (확인 불필요)**
- 이미 판별된 분기 구조를 화면 번호 체계로 반영하는 작업 (6절)
- 컬럼/필드 등 구조적 요소를 원본 소스와 대조해 발견된 차이를 채워 넣는 작업 (예: HTML에는 있는데 baseline엔 없는 컬럼 추가)
- 색상 매칭, 스타일 ID 바인딩, 컴포넌트 rename 여부 판단 등 이미 정해진 원칙의 적용
- 다음 단계가 명백히 이전 단계의 연장인 경우

**확인 필요**
- 0절의 시작 전 확인 항목(아직 값을 모르는 경우)
- 마스터 가이드에 참고할 것이 전혀 없어 신규로 만들어야 하는 경우 (1-2절 5순위)
- 개발팀 컨벤션처럼 이 문서에 원칙이 아직 없고 담당자만 알 수 있는 정보가 필요한 경우
- 마스터 가이드·프로토타입 코드 둘 다로 시각 규칙 판단이 안 되어 라이브 웹 확인이 필요한 경우 (스크린샷 요청)
- 작업 범위 자체를 넓히는 결정 (예: 이번 스코프 밖 화면까지 만들지 여부)

판단이 애매하면, "이미 합의된 원칙의 적용인가, 아니면 새로운 판단이 필요한가"를 기준으로 삼는다. 전자면 진행하고 결과를 보고하며, 후자만 진행 전에 확인한다.
